# Software Bill of Materials (SBOM) Guide

This guide explains the enhanced SBOM generation and security analysis workflow.

---

## 🎯 Overview

The Enhanced SBOM workflow provides comprehensive software supply chain transparency and security analysis:

- **Multi-Format SBOMs**: SPDX 2.3 and CycloneDX 1.6
- **Multi-Ecosystem Coverage**: .NET, npm, Docker
- **Vulnerability Scanning**: Grype, Trivy, OSV Scanner
- **License Compliance**: Automated license detection and validation
- **SBOM Attestation**: Cryptographic signing with Cosign
- **PR Integration**: Automatic security reports on pull requests

---

## 🚀 Quick Start

### Trigger SBOM Generation

The workflow runs automatically on:
- Push to `main` or `development` branches
- Pull requests to `main` or `development`
- New releases
- Manual trigger via GitHub Actions UI

### Manual Trigger

```bash
# Via GitHub CLI
gh workflow run generate-sbom-enhanced.yml

# Via GitHub UI
Actions → "Enhanced SBOM Generation & Analysis" → Run workflow
```

## 📦 Generated Artifacts

Each workflow run produces:

### 1. SBOM Files
- `sbom-dotnet.json` - .NET dependencies (CycloneDX)
- `sbom-*-spdx.json` - SPDX format SBOMs
- `sbom-*-cyclonedx.json` - CycloneDX format SBOMs
- `sbom-npm-*.json` - npm dependencies
- `sbom-docker-*.json` - Docker image contents

### 2. Vulnerability Reports
- `*-vulnerabilities.json` - Grype scan results
- `trivy-results.sarif` - Trivy scan (uploaded to GitHub Security)
- `osv-results.json` - OSV Scanner results
- `VULNERABILITY-SUMMARY.txt` - Human-readable summary

### 3. License Reports
- `npm-licenses-*.json` - npm license analysis
- `LICENSE-COMPLIANCE.md` - Compliance summary

### 4. Attestations
- `*.bundle` - Cosign signature bundles (if attestation enabled)

## 🔍 Understanding SBOM Formats

### SPDX (Software Package Data Exchange)
- **Standard**: ISO/IEC 5962:2021
- **Use Case**: Legal compliance, license tracking
- **Format**: JSON, YAML, RDF, Tag-Value
- **Tools**: Microsoft SBOM Tool, SPDX Tools

### CycloneDX
- **Standard**: OWASP standard
- **Use Case**: Security analysis, vulnerability tracking
- **Format**: JSON, XML
- **Tools**: CycloneDX .NET, npm plugin, Syft

## 🔐 Vulnerability Scanning

### Severity Levels

The workflow scans for vulnerabilities at these severity levels:
- **CRITICAL**: Immediate action required
- **HIGH**: Priority fix required
- **MEDIUM**: Should be addressed

### Scanners Used

#### Grype
- **Strength**: Fast, comprehensive CVE database
- **Coverage**: OS packages, language packages, Docker images
- **Output**: JSON, SARIF, table formats

#### Trivy
- **Strength**: Wide ecosystem support, IaC scanning
- **Coverage**: OS packages, language deps, config files
- **Integration**: Results uploaded to GitHub Security tab

#### OSV Scanner
- **Strength**: Google's open-source vulnerability DB
- **Coverage**: npm, PyPI, Go, Rust, etc.
- **Features**: Transitive dependency analysis

## 🔏 SBOM Attestation & Signing

### What is Attestation?

SBOM attestation cryptographically proves:
1. **Authenticity**: SBOM was generated by your CI/CD
2. **Integrity**: SBOM hasn't been tampered with
3. **Provenance**: SBOM is linked to specific commit/build

### How It Works

```
1. Workflow generates SBOM
   ↓
2. Cosign signs SBOM using GitHub OIDC (keyless)
   ↓
3. Signature bundle (.bundle file) created
   ↓
4. Attestation uploaded to GitHub
```

### Verify SBOM Signature

```bash
# Install Cosign
curl -sSfL https://raw.githubusercontent.com/sigstore/cosign/main/install.sh | sh

# Download SBOM and bundle from GitHub Actions artifacts
# Then verify:
cosign verify-blob sbom-dotnet.json \
  --bundle sbom-dotnet.json.bundle \
  --certificate-identity-regexp "https://github.com/your-org/.*" \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com
```

## 🛠️ Customization

### Add New Ecosystem

Edit `.github/workflows/generate-sbom-enhanced.yml`:

```yaml
# Add Python/pip support
- name: Generate Python SBOM
  run: |
    pip install cyclonedx-bom
    cyclonedx-py -o sbom-outputs/sbom-python.json
```

### Add New Scanner

```yaml
- name: Run Snyk Scan
  uses: snyk/actions/node@master
  with:
    args: --all-projects --sarif-file-output=snyk.sarif
```

### Custom Notification

```yaml
- name: Notify on Critical Vulnerabilities
  if: steps.grype-scan.outputs.critical_count > 0
  uses: slackapi/slack-github-action@v1
  with:
    webhook-url: ${{ secrets.SLACK_WEBHOOK }}
    payload: |
      {
        "text": "🚨 Critical vulnerabilities found in ${{ github.repository }}"
      }
```

## 🚨 Troubleshooting

### SBOM Generation Fails

**Problem**: CycloneDX tool not found
```
Solution: Ensure .NET SDK 10.0 is installed
dotnet tool install --global CycloneDX
```

**Problem**: npm SBOM fails
```
Solution: Check package-lock.json exists
npm install  # Regenerate lockfile
```

### Vulnerability Scan Fails

**Problem**: Grype rate limited
```
Solution: Add GITHUB_TOKEN to increase rate limits
```

**Problem**: False positives
```
Solution: Add to .trivyignore or grype.yaml
```

### Attestation Fails

**Problem**: "permission denied" error
```
Solution: Ensure workflow has id-token: write permission
```

## 📚 Best Practices

### 1. **Review SBOMs Regularly**
- Download and review SBOMs from releases
- Track dependency changes over time
- Audit new dependencies in PRs

### 2. **Address Vulnerabilities Promptly**
- CRITICAL: Fix within 24 hours
- HIGH: Fix within 1 week
- MEDIUM: Fix within 1 month

### 3. **License Compliance**
- Review all new dependencies for license compatibility
- Avoid GPL in proprietary code
- Document license decisions

### 4. **Keep SBOMs Updated**
- Regenerate on every release
- Archive historical SBOMs
- Version SBOMs alongside releases

### 5. **Automate Where Possible**
- Enable automatic PR comments
- Set up Slack notifications for critical issues
- Use Dependabot for automated updates
